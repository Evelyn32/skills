# 访谈提纲生成技能 (Interview Guide Generator)

## 设计目标

根据研究方案中的访谈阶段信息，自动生成结构完整、问题科学、可直接用于实施访谈的提纲文档。每次使用应交付一份包含开场/话题模块/收尾/探询指南的完整访谈提纲，同时提供结构化 JSON 供下游消费。

---

## 核心概念：两层技术体系

### 嵌入技术（来自研究方案，决定提纲中特定模块的组织方式）

嵌入技术是研究方案中标注的、依附于用户访谈的方法。它们各自作用于提纲中**对应的话题模块**，而非整场访谈的全部内容：

| 嵌入技术 | 在提纲中的作用 | 识别信号 |
|---------|--------------|---------|
| **JTBD** | 当提纲中有涉及用户选择/切换/购买决策的话题模块时，用 JTBD 的决策链逻辑（触发→竞争→决策→评估）组织该模块的问题 | 研究方案标注了 JTBD，或用户提到"选择""放弃""竞品""决策" |
| **灵犀旅程** | 当提纲中有涉及端到端体验的话题模块时，用旅程回溯法（行为序列→触点→情绪曲线）组织该模块的问题 | 研究方案标注了灵犀旅程，或用户提到"旅程""触点""全过程" |
| **卡片分类** | 在提纲话题模块之后增加一个独立的分类任务模块（15-20分钟），获取用户的心智分组模型 | 研究方案标注了卡片分类，或用户提到"信息架构""导航分类" |

> **关键理解**：一场访谈的提纲由多个话题模块组成，JTBD 或灵犀旅程只影响其中部分模块的组织方式，而非整场访谈的框架。其他模块（如认知画像、使用习惯等）仍按标准的 CBA 方式组织。卡片分类是一个独立的任务模块，插在话题模块之后。

### 题目技术（始终可用，无需研究方案标注）

这些技术用于**所有模块**的题目设计和访谈执行，是提纲的"基础设施"：

| 题目技术 | 作用 | 在提纲中的位置 |
|---------|------|--------------|
| **CBA 原则** | 题目设计：确保每个话题从情景→行为→态度三个角度全面覆盖 | 贯穿所有模块的题目编写 |
| **ORID** | 追问深度：在需要深挖某个话题时，按事实→感受→思考→行动逐层追问 | 探询指南章节 |
| **360度提问法** | 对话方向盘：访谈中实时判断该上推总结、平移换角度、还是下切追问 | 探询指南章节 |
| **投射技术** | 间接探测：当用户难以直接表达时，用拟人/联想/第三人称等方式获取真实想法 | 探询指南章节 |

> **CBA 不等同于"框架"**：CBA 是题目设计法，用于决定每个问题怎么写，而不是决定提纲分几个模块。提纲要分哪几个话题模块，由研究方案的 content_outline 决定。

## 使用流程

### 输入信息

从体验伙伴 Agent 接收（或用户直接提供）以下信息：

- **研究目标**：访谈要回答的核心问题
- **目标用户**：被访者特征（角色、经验、使用频率等）
- **产品/服务背景**：是什么产品、当前状态
- **研究方案内容概览**（优先使用）：研究方案中访谈阶段的话题模块、核心探针、嵌入技术标注。这是模块划分的主要依据
- **前置研究结论**（可选）：已有数据、问卷发现、走查结果
- **约束条件**（可选）：单场时长限制、执行形式（线上/线下）

若信息不足，主动追问补齐。若无研究方案提供 content_outline，根据研究目标自行拆解话题维度。

### Step 1: 解析输入

#### 1.1 确定话题模块

话题模块是提纲的主干。优先从研究方案的 content_outline 获取；若无，则根据研究目标自行拆解：

- 从研究目标中提取 3-5 个核心话题维度
- 维度顺序遵循：简单→复杂、具体→抽象、行为→态度
- 每个话题模块化约为 10-20 分钟

#### 1.2 识别各模块适用的嵌入技术

检查研究方案中该访谈阶段是否标注了嵌入技术，并按模块匹配：

| 嵌入技术 | 匹配哪个模块 | 如何处理 |
|---------|------------|---------|
| **JTBD** | 涉及用户选择/切换/决策的话题模块 | 该模块内部用 JTBD 的决策链逻辑组织问题（触发→竞争→决策→评估），但不影响其他模块 |
| **灵犀旅程** | 涉及端到端体验/旅程的话题模块 | 该模块内部用旅程回溯法组织（行为序列→触点→情绪变化），可融入对应话题模块或作为独立模块 |
| **卡片分类** | 独立任务模块，不属于任何话题模块 | 在所有话题模块之后插入（15-20分钟） |

> 如果研究方案未标注任何嵌入技术，所有模块默认用 CBA 方式设计题目。

#### 1.3 读取参考文件

- 始终读取：`references/cba-framework.md`、`references/probing-techniques.md`
- 有 JTBD 模块时读取：`references/jtbd-framework.md`
- 有灵犀旅程模块时读取：`references/journey-mapping.md`
- 有卡片分类时读取：`references/card-sorting.md`

### Step 2: 提纲结构设计

#### 2.1 通用结构骨架

所有访谈提纲都遵循四段结构：

```
一、开场（约5分钟）
   ├─ 自我介绍 + 项目背景
   ├─ 录音知情同意
   └─ 暖场寒暄

二、话题模块（约35-50分钟）
   ├─ 模块一：[话题名称]（约X分钟）  ← 标注组织方法：标准CBA / JTBD组织 / 旅程回溯
   ├─ 模块二：[话题名称]（约X分钟）
   ├─ ...
   └─ 卡片分类任务模块（如有，约15-20分钟）

三、收尾（约5分钟）
   ├─ 查缺补漏
   ├─ 补充询问
   └─ 感谢 + 报酬说明

四、探询指南（始终输出）
   ├─ 追问三向仪（360度提问法速查）
   ├─ ORID 追问链
   ├─ 投射技术速查
   └─ 特殊用户应对策略
```

#### 2.2 时长分配

| 配置 | 总时长 | 开场 | 话题模块 | 嵌入任务 | 收尾 |
|------|-------|------|---------|---------|------|
| 纯话题模块（无嵌入任务） | 45-60分钟 | 5分钟 | 35-50分钟 | - | 5分钟 |
| 含卡片分类 | 60-75分钟 | 5分钟 | 35-45分钟 | 15-20分钟 | 5分钟 |
| 探索性较强 | 60-75分钟 | 5分钟 | 50-65分钟 | - | 5分钟 |

#### 2.3 模块内部组织：按话题特点选择方法

每个话题模块内部如何组织题目，取决于该模块的话题特点：

**标准 CBA 组织**（默认，适用于大多数模块）：按 "情景 → 行为 → 态度" 的递进关系组织题目序列。

```
模块：品牌认知（约10分钟）

C 类铺垫："提到银行品牌，你第一个想到的是……"
B 类展开："你这么看的原因是什么？"
A 类收束："你会向别人推荐吗？怎么推荐？"
```

**JTBD 组织**（当模块涉及选择/决策时）：按决策链逻辑组织题目序列。

```
模块：理财选择决策（约15分钟）[嵌入技术：JTBD]

情境还原："你上次决定买理财产品是什么情况？"
Trigger："是什么让你觉得需要做点什么？"
竞争方案："你还考虑了哪些选择？"
决策："最后让你下决心的那一下是什么？"
使用评估："用了一段时间后，和你预期的比怎么样？"
```

**旅程回溯组织**（当模块涉及端到端体验时）：按行为链+情绪标注组织题目序列。

```
模块：手机银行使用全过程（约15分钟）[嵌入技术：灵犀旅程]

行为序列："能一步一步描述你刚才操作的过程吗？"
触点情绪："每一步的感觉有什么不同？"
高低点："哪一步最顺畅？哪一步最想放弃？"
痛点追问："你觉得为什么会在这一步卡住？"
```

#### 2.4 卡片分类任务模块

> **前提**：本技能仅处理卡片分类嵌入访谈的场景（默认）。当研究方案中卡片分类作为独立阶段执行时（样本量≥15人、在线无主持工具），由研究方案直接设计，不通过本技能生成。

| 环节 | 时长 | 内容 |
|------|------|------|
| 任务说明 | 2分钟 | 解释规则：按自己的理解分组，没有对错 |
| 开放式分类 | 10分钟 | 用户自由分组并给每组命名 |
| 出声思考 | 伴随 | 记录分组逻辑和疑惑点 |
| 结果确认 | 3分钟 | 复述用户分组，确认理解 |

### Step 3: 问题编制

#### 3.1 CBA 题目设计法（所有模块默认使用）

无论话题模块采用哪种组织方法，**每个模块内的具体题目都使用 CBA 原则设计**：

| 角度 | 含义 | 典型问法模板 |
|------|------|------------|
| **C — 情景** | 行为发生的情境、背景 | "当时是在什么情况下……""能描述一下那天的场景吗？" |
| **B — 行为+原因** | 做了什么、为什么 | "你是怎么做的？""是什么原因让你选择这个？" |
| **A — 态度+建议** | 怎么看、希望怎么解决 | "你对这个怎么看？""你希望怎么改进？" |

每个模块建议 2-4 道核心题目，每道题目配 1-3 个追问方向。

#### 3.2 题目设计原则

| 原则 | 说明 | 正例 | 反例 |
|------|------|------|------|
| **锚定具体事件** | 让用户回到真实经历中 | "上一次你转账时……" | "你平时怎么转账？" |
| **多问开放问题** | 不预设答案方向 | "当时是什么感觉？" | "是不是觉得挺方便的？" |
| **侧面迂回** | 一个目标拆为多个问题间接询问 | 分三个角度问办卡原因 | "你为什么要办这张卡？" |
| **一题一问** | 避免连珠炮 | "你选择它的原因是什么？" | "你为什么选它、什么时候用的、感觉怎么样？" |
| **不预设前提** | 不假设用户一定有某行为 | "你有过理财经历吗？" | "你上次买理财时……" |
| **投石问路** | 先铺垫再深入 | 先问"第一张卡是哪家的"再问"当时怎么办的" | 直接问"你的办卡决策因素是什么" |

#### 3.3 禁忌题目（必须避免）

| 禁忌类型 | 示例 | 正确替代 |
|---------|------|---------|
| **否定性假设**（隐含指责） | "你为什么没有用我们的产品？" | "目前有哪些方式在帮你解决这个问题？" |
| **未来预测**（不准确） | "未来你会考虑用这个产品吗？" | "现在有没有类似需求？你是怎么解决的？" |
| **让用户定价**（超出能力） | "你觉得费率多少合适？" | "费用在你的考虑中排第几位？为什么？" |
| **让用户设计**（超出能力） | "你觉得应该设计成什么样？" | "你在使用中遇到了什么问题？" |
| **双重问题** | "你觉得功能和界面怎么样？" | 拆成两道题分别问 |
| **含价值判断** | "你觉得这个好用的功能……" | "你使用这个功能的过程中……" |

#### 3.4 问题呈现格式

每个模块的题目以表格形式呈现：

```
| 铺垫问题 | 核心问题（追问） | 设置目的 | CBA类型 |
|---------|----------------|---------|--------|
| [从简单、具体切入] | [核心探针 + 追问链] | [映射到研究目标] | C/B/A |
```

JTBD 组织的模块可在表格上方注明 "本模块采用 JTBD 决策链逻辑组织"，旅程回溯模块同理。

### Step 4: 探询指南生成

探询指南始终输出，供研究员在访谈过程中随时参考：

#### 4.1 追问三向仪（360度提问法速查）

| 方向 | 含义 | 适用时机 | 示例 |
|------|------|---------|------|
| **上推** | 总结归纳、提炼高度 | 用户讲了大量细节后 | "您分享的这些，如果用一句话概括……" |
| **平移** | 换个角度、举一反三 | 话题卡住或需要对比 | "除此之外，还有其他类似的情况吗？" |
| **下切** | 追问细节、层层剥开 | 发现新线索需要深挖 | "为什么？""是什么原因让您这样觉得？" |

#### 4.2 ORID 追问链

| 层级 | 问题类型 | 典型问法 |
|------|---------|---------|
| O — 事实 | 发生了什么 | "你看到了什么？当时页面是什么样的？" |
| R — 感受 | 什么感觉 | "你当时什么感觉？是着急还是无所谓？" |
| I — 思考 | 你怎么看 | "你觉得为什么会这样？这对你意味着什么？" |
| D — 行动 | 你会怎么做 | "你希望怎么改进？如果让你自己设计呢？" |

#### 4.3 投射技术速查

| 技术 | 用法 | 示例 |
|------|------|------|
| **拟人法** | 把产品/品牌想象成人 | "如果把工行想象成一个人，他是什么样的性格？" |
| **联想法** | 快速联想 | "提到手机银行，你第一个想到的词是什么？" |
| **第三人称法** | 借他人之口 | "你觉得你室友为什么不用这个功能？" |
| **完成法** | 句子填空 | "银行 App 最让我受不了的是____" |

#### 4.4 特殊用户应对

| 用户类型 | 特征 | 应对策略 |
|---------|------|---------|
| **好奇用户** | 反问研究员、征求意见 | 坚持倾听者定位："您的情况我很感兴趣，能再多说说吗？" |
| **专家用户** | 给设计方案、技术建议 | 不过分关注方案本身，追问方案解决了什么问题 |
| **佛系用户** | "都挺好""无所谓" | 用对比制造思考张力："有人觉得A重要，有人说B，你偏向哪边？" |

### Step 6: 质量检查

生成提纲后逐项检查：

**嵌入技术**
- [ ] 各模块是否标注了组织方法（标准CBA / JTBD / 旅程回溯）？
- [ ] 嵌入技术是否只作用于对应的话题模块，而非整场访谈？
- [ ] 卡片分类任务模块是否在所有话题模块之后？

**题目层面**
- [ ] 是否存在禁忌题目（否定假设、未来预测、让用户定价/设计）？
- [ ] 核心题目是否以开放题为主？
- [ ] 每个核心题目是否配了追问方向？
- [ ] 是否避免了双重问题和引导性措辞？
- [ ] 题目是否锚定具体事件（而非抽象态度）？
- [ ] 每个题目能否追溯到研究目标？

**结构层面**
- [ ] 模块顺序是否符合"简单→复杂、具体→抽象、行为→态度"？
- [ ] 时长分配是否合理（开场≤5min，收尾≤5min）？
- [ ] 开场是否含自我介绍、录音同意、暖场三个环节？
- [ ] 收尾是否含查缺补漏、补充询问、感谢+报酬说明？
- [ ] 探询指南是否完整（追问三向仪 + ORID + 投射技术 + 特殊用户）？

**可执行性**
- [ ] 被访者画像是否清晰可筛？
- [ ] 物料准备是否列全？
- [ ] 记录表模板是否提供？

## 多场次/多角色策略

- **默认**：一份提纲覆盖所有被访者
- **角色差异大时**：话题模块保持相同，开场/暖场按角色微调
- **多场次迭代**：前几场完之后可根据发现微调后几场的追问重点，但提纲主体不变

## 与上下游技能协作

- **← research-plan-generator**：研究方案中的访谈阶段（content_outline、embedded_techniques、sample_characteristics）作为输入
- **← problem-clarifier**：澄清后的研究问题陈述作为目标输入
- **→ 访谈实施**：提纲直接交付研究员执行
- **→ 结果分析**：提纲中的维度-问题追溯表可反向辅助结果编码

## 特殊场景处理

| 场景 | 处理方式 |
|------|---------|
| 无研究方案直接使用 | 先追问目标、用户、背景三要素，自行拆解话题维度，默认用 CBA 组织 |
| B 端产品访谈 | 注重决策链角色覆盖（决策者/使用者/影响者），常需 JTBD 组织决策相关模块 |
| 对新手用户访谈 | 题目更加具体和场景化，减少抽象概念，暖场时间适当延长 |
| 单场时长受限（≤30分钟） | 聚焦 1-2 个核心话题，裁剪嵌入任务和次要追问链 |
| 线上线下混用 | 线上注意保留更多追问空间，线下可增加互动任务 |
| 有前置问卷数据 | 将问卷发现作为铺垫问题的引子（"从问卷中我们看到……你的情况是？"） |

## 常见错误提醒

- 不要把提纲写成问卷——访谈问题是探针，不是选择题
- JTBD 和灵犀旅程不是整场访谈的框架，只作用于对应的话题模块
- CBA 不是框架，是题目设计法——不要用 CBA 分模块，而是在每个模块内用 CBA 写题目
- 嵌入技术模块（卡片分类）在嵌入模式下不能独立存在——必须与主访谈模块共享同一批被访者
- 禁忌题目检查不能省——尤其是"让用户设计"和"未来预测"这两类高频错误
- 探询指南不是可选附录——是访谈员手中的导航仪，必须输出
- 话题模块划分优先遵循研究方案的 content_outline，不要自己另起炉灶
- 题目之间要有逻辑递进——不是话题罗列，而是一根根探针循序渐进地深入
