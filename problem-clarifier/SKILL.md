# 问题澄清技能 (Problem Clarifier)

## 核心理念

帮用户把一句模糊的话变成一句清晰的研究问题。每个环节让用户做**可多选的勾选题**，同时支持自定义填写，不需要写大段文字。

## 何时触发

**触发条件**：用户输入存在以下任一特征时主动介入：

| 信号 | 示例 |
|------|------|
| 研究目标过于宽泛 | "用户还有什么需求" |
| 核心概念模糊未定义 | "体验怎么样" |
| 多个问题混在一起 | "满意度和使用习惯和新功能接受度" |
| 只给了动作没有方向 | "帮我调研一下这个产品" |

**不触发**：用户输入已经包含清晰的研究对象 + 目标人群 + 研究意图，则直接交给 Research Plan Generator，不画蛇添足。

**不触发的示例**：
- "评估新版首页的信息架构是否合理，面向活跃用户"
- "了解新注册用户对引导流程的满意度"

## 交互方式

**每轮使用 `question` 工具与用户交互**，设置 `multiple: true` 允许用户多选，同时启用 `custom: true` 支持用户自行填写选项中没有的答案。

每个选项使用 `☐` 前缀表示可多选。选项末尾始终包含 `☐ 跳过此问题` 和 `☐ 其他（请描述）`。

## 追问策略

核心原则：**不按固定剧本走**。每轮先看用户已经说了什么，找出信息缺口，挑最重要的那个问。像一个有经验的用研人员在对话中逐步聚焦，而不是填一张标准化表格。

### 需要澄清的五个维度（非固定顺序）

每次对话，判断以下维度哪些已明确、哪些还是缺口：

| 维度 | 要搞清楚 | 何时必须问 |
|------|---------|-----------|
| **研究方向** | 要研究什么？满意度、痛点、需求、易用性、行为、态度…… | 用户表述模糊时必问 |
| **研究动机/决策背景** | 为什么研究？结论用来做什么决策？ | 研究目标涉及"改版""提案""方案对比"等决策场景时必问 |
| **目标人群** | 面向谁？是否区分经验/角色/频率？ | 用户未明确提及人群特征时必问 |
| **聚焦范围** | 全产品还是某个模块/流程/环节？ | 研究对象可以进一步聚焦时必问 |
| **约束条件** | 时间、渠道、样本等有无特殊要求？ | 有明显时间压力或渠道限制时才问 |

### 每轮的操作方式

1. **回顾用户已有输入**，标记五个维度中哪些已明确、哪些还是缺口
2. **从缺口中挑最重要的一个**，用 `question` 工具发问（`multiple: true`，`custom: true`）
3. **用户回答后，先做一句话总结确认**（"好的，我理解你想……对吗？"），再更新缺口清单
4. **重复**，直到剩余缺口不影响研究方案生成为止

### 什么情况下要追问

不要因为用户选了"跳过"就放弃。以下情况主动追下去：

- 用户说的话仍然比较泛，需要进一步聚焦（如"体验不好"→ 哪个环节？谁的体验？在什么场景下？）
- 用户跳过了某个可能影响方案设计的维度，AI 判断后主动追问
- 用户没有提到但明显有必要的维度——比如调研目的是"为改版提案做准备"，但没有说时间要求
- 用户输入中出现歧义或多种理解方式，给选项让他选

### 什么情况下可以少问

- 用户的初始输入已经覆盖了大部分维度，只剩一两个缺口
- 用户在对话中表现出"不想纠结细节，你帮我定"的态度
- 剩余的缺口是研究方案层能自行处理的（如不指定样本量，留给方案层）

### 什么时候停止

- 五个维度中影响方案设计的关键信息都已明确
- 用户明确表示"就这些了，不需要再细化"
- 再问下去边际收益很低时，把剩余模糊点交给 Research Plan Generator 自带处理

### 产品阶段对照表（场景匹配时才用）

当用户的研究对象是一个产品/服务且存在明确生命周期时，可以问其所处阶段作为上下文参考。阶段对应的典型研究方向见 `references/question-templates.md`。

如果用户的研究对象不是产品（如内部工具、流程、服务），不强行套用。

**阶段的作用**：仅用于在生成研究方向选项时把推荐方向排在前面，不做过滤——运营阶段也可能做需求调研。

## 追问原则

1. **每轮使用 `question` 工具，`multiple: true`，`custom: true`**，让用户勾选或自行填写。每个选项使用 `☐` 前缀
2. **一次只问一个核心维度**，可附带最多一个简单的补问（如选了"具体功能"没说是哪个 → 同轮追问"哪个功能？"）
3. **用户回答每一轮后，先做一句话总结确认**，再进入下一轮
4. **用户说"不确定"或"跳过"时，不等他纠结，先给默认建议推进；但如果该维度确实关键，换种问法再试一次**
5. **不设轮数上限**，以"缺口填够"为准。大多数情况 3-5 轮收工
6. **已经明确的维度不回头追问**
7. **模板是参考案例，不是必须执行的脚本**。遇到模板覆盖不到的场景，根据用户原话自由设计选项。核心约束不变：多选 + 自定义 + 跳过
8. **阶段对照表只用来排序选项（推荐放前面），不做选项过滤**

## 输出格式

澄清完成后，输出以下内容：

```markdown
## 研究问题陈述

[用一句自然语言概括本次研究的核心问题。包含：研究对象 + 目标人群 + 研究意图。]

---

以上理解是否正确？确认后我将为你生成研究方案。
```

**输出示例**：

```
## 研究问题陈述

活跃用户（近30天有使用行为）对当前产品还有哪些未被满足的功能需求？（探索性研究，全产品范围）

---

以上理解是否正确？确认后我将为你生成研究方案。
```

输出后，如用户确认无误，将研究问题陈述交给 Research Plan Generator 继续。

## 失败处理

- 如果多轮后用户仍觉得不清晰，输出当前最接近的解读并标注"存在不确定性"，建议直接用访谈方式做开放式探索
- 如果用户输入完全超出预设选项框架，转为简短的自由追问，不做选择题
