# 灵犀旅程 (UDS)

> **全称**：灵犀旅程——基于用户旅程的需求痛点分析法（UDS，User Journey for Demand Pain Points Analysis Scheme）
>
> **来源**：ICBC 中国工商银行 业务研发中心

---

## 一、方法定义

灵犀旅程（UDS）是一个标准化的研究分析方法，通过站在用户视角结合用户行为与产品触点，系统性地分析用户需求和痛点，定位关键旅程环节，形成可量化的体验优化方案。

- **输入要素**：用户思维、用户态度、用户需求、用户声音、用户行为、业务目标、业务流程、操作页面等
- **输出成果**：用户旅程图 + 优化建议列表
- **资源投入**：完成一次旅程分析约需 **3-4 人/天**

### 使用基本条件

- 产品中有明确的操作或使用用户（客户）
- 用户（客户）与产品具有明确的交互流程与步骤

---

## 二、适用场景

灵犀旅程适用于三大类场景：

| 场景类型 | 说明 | 侧重点 |
|---------|------|--------|
| **市场首创** | 市场中尚无同类产品，我行首次创新 | 创意阶段（了解用户行为模式）→ 需求编写（预测潜在痛点）→ 运营阶段（收集真实用户声音） |
| **行内创新** | 市场有同类产品，我行尚未建立 | 立项阶段（汇总竞品优势）→ 需求方案 → 运营阶段（持续提升） |
| **产品优化** | 产品已投放市场，需优化提升 | 需求方案（分析现有痛点）→ 需求编写 → 运营阶段（前后对比评估效果） |

**信号关键词**：旅程、触点、端到端、全过程、使用路径、体验地图、需求痛点、全流程分析

## 三、用户旅程图结构

旅程图是灵犀旅程方法的核心产出物，以结构化形式组织五项信息：

```
旅程图
├── 旅程结构        — 场景、行为、触点（含角色拆分）
├── 驱动因素        — 用户侧需求偏好 + 银行侧业务目标
├── 体验指标        — 主观态度类、行为交易类、声音问题类
├── 典型问题        — 与驱动因素相关的用户体验问题（按优先级排序）
└── 优化点          — 针对典型问题的具体优化建议
```

### 3.1 旅程结构

将目标场景逐层分解为 **场景 → 行为 → 触点**：

| 层级 | 定义 | 示例（手机银行购买理财） |
|------|------|---------------------|
| **场景** | 用户使用产品的主要目标和操作渠道 | "手机银行购买理财" |
| **行为** | 达成场景的具体操作步骤名称 | "查找产品 → 查看详情 → 确认购买 → 完成支付" |
| **触点** | 用户与产品交互的每个具体页面/环节 | 产品列表页、详情页、确认页、支付页 |

触点可进一步按角色拆分（如用户触点 vs 柜员触点），便于跨角色分析。

### 3.2 驱动因素

驱动因素是促使用户在各行为下获得良好体验的核心需求，分为用户侧和产品侧。

**用户侧驱动因素**：用户在各行为下的核心需求与偏好，通过**用户访谈、体验问题分析、工作坊**等形成，以"形容词+名词"形式表述，如"信息准确清晰""操作便捷流畅"。

常用参考维度：功能完善有用、操作流畅便捷、信息完整准确、使用安全有保障、视觉美观愉悦、服务智能个性化、服务专业有效、运营内容丰富。

**产品侧驱动因素**：对应于用户行为的业务目标与诉求。

> 每个行为下只保留最重要的驱动因素。若某行为下驱动因素过多，说明行为划分过粗。

### 3.3 体验指标

分为三类：

| 指标类型 | 示例 | 数据来源 |
|---------|------|---------|
| **主观态度类** | ETS分值、功能流程满意度、交互设计评分等 | ETS评估模型、用户问卷 |
| **行为交易类** | 操作时间、转化率、交易量、系统响应速度 | 数据看板、灵活查询 |
| **声音问题类** | 投诉数量、各行为声音问题数量、情感指数 | 用户声音文本分析 |

指标选择原则：基于驱动因素制定多个候选指标 → 圈定重点指标 → 结合获取成本和干系人意见最终选定。如确实缺少某类指标，可暂不列出但需推动数据获取。

### 3.4 典型问题

通过多渠道收集与驱动因素相关的用户体验问题：体验管理平台优化建议、工银众测、95588工单、用户访谈、可用性评估等。收集后按以下规则处理：

- 筛选：原则上须与驱动因素有关，确实重要的问题可酌情纳入
- 整理：将原始问题分类汇总为典型问题，对应至具体行为或触点
- 排序：按重要度分为一级/二级/三级，一级为最优先

### 3.5 优化点

基于典型问题优先级和对应驱动因素，形成简洁、明确的优化建议：

```
优化优先级 = 关键触点一级问题 → 关键触点二级问题 → 非关键触点一级问题 → ...
```

优化建议应录入用户体验管理平台追踪管理。

## 四、实施步骤

灵犀旅程包含五个步骤：

| 步骤 | 名称 | 说明 |
|------|------|------|
| 1 | 梳理旅程图 | 确定对象 → 形成旅程结构 → 挖掘驱动因素 → 定制数据指标 → 绘制旅程图框架 |
| 2 | 数据前测 | 收集体验指标数据 → 补充典型问题 → 形成优化点 |
| 3 | 体验提升 | 将优化点纳入研发需求，推动产品改进 |
| 4 | 数据后测 | 上线至少 1 个月后再次采集数据，与前测对比，验证提升效果 |
| 5 | 持续更新 | 产品更新后修改调整旅程图，持续发挥评估价值 |

### 关键操作细节

**步骤1 梳理旅程图：**
- 先确认评估对象是否满足基本条件（有明确用户 + 有明确交互流程）
- 通过用户访谈、体验问题分析等定性方法形成驱动因素
- 驱动因素需用**用户视角语言**表述（如"办理效率高"而非"缩短操作时长"）

**步骤2 数据前测：**
- 关键触点识别逻辑：体验指标数值低点 + 典型问题集中出现的触点 = 关键环节

**步骤4 数据后测：**
- 数据获取方式、调研对象原则上须与前测保持一致
- 输出指标变化值和前后问题数量对比

## 五、推荐样本量

灵犀旅程的核心数据来源是**定性输入 + 既有量化数据**，因此：

| 数据来源 | 推荐样本量 | 说明 |
|---------|-----------|------|
| 用户访谈（形成驱动因素和典型问题） | 8-20 人 | 覆盖不同用户类型和使用场景，以信息饱和为原则 |
| 既有量化数据（体验指标） | 全量 | 来自数据看板、ETS评估模型、用户声音分析等已有数据源 |
| 数据后测（对比验证） | 与数据前测一致 | 采集方式和对象原则上保持不变 |

通常不需要为灵犀旅程单独大规模采集数据——它强依赖于**整合既有数据源**（ETS数据、行为日志、声音分析），访谈部分与主采集方式合并执行即可。

## 六、数据采集归属

灵犀旅程在研究方案的方法层级体系中具有特殊定位：

- **嵌入技术**：旅程结构的梳理和驱动因素的挖掘可作为用户访谈阶段的一部分——访谈中引导用户回溯完整体验旅程（15-20 分钟），使用同一批被访者
- **综合产出**：完整的旅程图需要结合访谈定性发现 + ETS定量数据 + 行为交易数据 + 用户声音数据等多数据源整合，本质上是一个综合分析方法

### 在研究方案中的典型用法

| 场景 | 推荐用法 |
|------|---------|
| 探索型研究 + 旅程分析 | 用户访谈阶段中嵌入灵犀旅程——访谈话题覆盖行为回溯、驱动因素探索和典型问题收集，后续整合量化数据绘制完整旅程图 |
| 评估型研究 + 旅程分析 | 在 ETS 评估或可用性评估后追加灵犀旅程分析——利用既有量化数据作为体验指标，辅以少量补充访谈收集典型问题和优化方向 |
| 综合诊断型 | 灵犀旅程作为串联多数据源的框架——整合用户访谈(驱动因素) + ETS(主观态度指标) + 行为数据(交易指标) + 用户声音分析(声音指标)，形成完整的五要素旅程图 |

## 与其他方法的衔接

- **← 用户访谈**：访谈数据作为驱动因素和典型问题的主要来源
- **← ETS体验评估**：ETS 分值作为主观态度类体验指标
- **← 用户声音分析**：声音数据作为声音问题类体验指标
- **← 可用性评估**：可用性问题纳入典型问题清单
- **→ 优化建议列表**：输出物可作为产品需求编写和研发优先级决策的输入

---

## 七、常见误区

- 把旅程图当成功能流程图画——缺少驱动因素和体验指标，变成纯操作步骤罗列
- 驱动因素从假设出发而非从真实用户数据出发
- 仅使用主观态度指标，忽略行为交易和声音问题指标（三类指标互补才能全面评估）
- 数据前测后不安排后测，无法量化体验提升效果
- 典型问题不做分类和优先级排序，无法聚焦关键改进方向
