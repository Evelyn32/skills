# 研究方案生成技能 (Research Plan Generator)

## 设计目标

根据用户的研究需求和提问，自动生成结构完整、方法论科学、可直接用于研究执行的研究方案文档。每次使用应输出一份包含研究方法、研究内容概览、样本量、样本特征和配额要求的完整方案，同时提供结构化 JSON 供下游消费。

---

## 方法层级体系

本技能支持的方法分为三个层级。必须理解这个层级关系，才能正确设计研究方案。

### 层级一：数据采集方式（主采集方式）

决定"如何接触用户并获取数据"，是研究方案的基本骨架。每个研究阶段以数据采集方式为锚点，独立执行（独立招募、独立时间窗口）。

| 数据采集方式 | 说明 | 典型执行形式 |
|-------------|------|-------------|
| **用户访谈** | 一对一深度对话，获取定性洞察 | 面对面 / 线上视频，45-75分钟/人 |
| **问卷调查** | 大规模量表/选择，获取定量数据 | 在线问卷平台投放 |
| **可用性评估** | 任务操作观察，获取行为数据 | 实验室 / 远程测试，40-60分钟/人 |
| **用户声音分析** | 已有反馈文本分析，获取自然数据 | API/爬虫采集 + NLP分析 |

### 层级二：嵌入分析技术

在数据采集过程中嵌入的特定分析工具或结构化框架。默认依附于某一数据采集方式执行，共享该阶段的用户和样本量。

| 嵌入技术 | 依附的主采集方式 | 说明 |
|---------|----------------|------|
| **卡诺模型** | 问卷调查 | 问卷中的功能属性分类题型（正向+反向双题） |
| **ETS体验评估** | 问卷调查 | 问卷中的体验诊断框架（ETS-U：NPS+双选 / ETS-E：48+26题） |
| **JTBD** | 用户访谈 | 在涉及用户选择/切换/决策话题的访谈模块中用决策链逻辑组织题目（触发→竞争→决策→评估） |
| **卡片分类** | 用户访谈（嵌入）<br>或 独立执行 | 访谈中嵌入的分类任务（默认）；当样本量需求超出访谈范围（≥15人）或采用在线无主持方式时，可按独立数据采集方式执行 |
| **眼动测试** | 可用性评估 | 可用性测试中的视线追踪数据采集方式 |
| **灵犀旅程** | 用户访谈 | 在涉及端到端体验话题的访谈模块中用旅程回溯法组织题目（行为序列→触点→情绪曲线） |

> **卡片分类的灵活定位**：卡片分类在方法本质上是独立的数据收集方法（有独有的研究问题、执行方式、样本量要求和分析方法），但在实践中几乎总是配合访谈使用（需出声思考获取分组逻辑），因此默认作为嵌入技术处理。当样本量要求超出访谈范围或采用在线无主持工具时，按独立阶段设计，样本量使用卡片分类自身标准（15-30人）。详见 `references/card-sorting.md` 中的判定规则。

### 层级三：综合产出

需要多个数据源整合分析的产出物，决定的是"最终交付什么"，本身不独立招募用户。

| 综合产出 | 数据来源 | 说明 |
|---------|---------|------|
| **用户画像** | 用户访谈 + 问卷调查 | 访谈建立行为假设 → 问卷聚类验证 → 形成Persona |
| **灵犀旅程** | 用户访谈（+ 用户声音分析） | 访谈回溯旅程 → 声音分析补充触点 → 绘制旅程地图 |

### 层级关系的核心原则

1. **方案以数据采集方式为阶段划分依据**：一个阶段 = 一种数据采集方式 + 若干嵌入技术
2. **嵌入技术合并到主采集方式内**：卡诺模型和ETS体验评估可以共享同一份问卷；卡片分类可以和需求访谈合并为同一场访谈的后半段
3. **阶段数 ≤ 4 个**：按数据采集方式计数，嵌入技术不限数量
4. **同阶段可执行多个目标**：一份访谈可以同时探索需求痛点 + 完成卡片分类 + 回溯旅程
5. **样本量按主采集方式计算**：合并执行时，以主采集方式的样本量为准
6. **卡片分类例外规则**：当卡片分类因以下原因不适合嵌入访谈时，可作为独立的数据采集方式阶段执行：
   - 样本量需求超出访谈范围（聚类分析稳定需 ≥15 人，访谈信息饱和通常 8-12 人）
   - 采用在线无主持工具执行（OptimalSort 等），不涉及访谈对话
   - 独立执行时，样本量使用卡片分类自身标准，阶段数计入 ≤4 限制

## 使用流程

### 输入信息

从用研智能体或用户接收以下信息：

- **研究目标**：想研究什么、解决什么问题
- **产品/服务背景**：是什么产品、当前状态（新功能/改版/现有问题诊断）
- **目标用户**：被访者特征（终端用户/专家/内部团队/潜在用户）
- **已有认知**（可选）：已有的数据、访谈发现、走查结果
- **约束条件**（可选）：预算、时间、可接触用户范围

若信息不足，主动向用户追问补齐。

### Step 1: 需求解析

从用户输入中提取以下结构化信息：

1. **研究类型判断**：
   - 探索型：理解用户行为、发现需求、定义问题空间
   - 测量型：量化现状、确定优先级、验证假设
   - 评估型：测试方案、诊断问题、比较方案
   - 描述型：刻画用户群体、建立认知基线

2. **研究子目标拆解**：将对用户的一句话需求拆解为具体的研究子目标（如"了解认知 + 挖掘需求 + 验证普遍性"）

3. **约束条件**：时间紧迫程度、可及样本规模、有无实验室设备

### Step 2: 层级式方法论匹配

按"先选采集方式 → 再嵌入技术 → 合并同类阶段"的顺序进行。

#### 2.1 主采集方式匹配

根据研究类型确定数据采集方式：

| 研究类型 | 主采集方式 | 判断依据 |
|---------|-----------|---------|
| 探索型 | **用户访谈** | 需要深入理解原因、动机、态度，定性数据 |
| 测量型 | **问卷调查** | 需要量化、统计、验证普遍性，定量数据 |
| 评估型（体验水平） | **问卷调查** | ETS体验评估需要大规模用户评分 |
| 评估型（交互界面） | **可用性评估** | 需要观察操作行为、任务绩效 |
| 描述型 | **用户访谈 + 问卷调查** | 先定性探索维度，再定量描述分布 |
| 已有数据利用 | **用户声音分析** | 需要从现有反馈文本中提取洞察 |

**混合型场景**：当用户需求覆盖多种类型时，分阶段组合不同的主采集方式。典型路径：
- 探索+测量 → 用户访谈(阶段一) → 问卷调查(阶段二)
- 评估(体验)+评估(交互) → 问卷调查(ETS,阶段一) → 可用性评估(阶段二)

#### 2.2 嵌入技术匹配

根据研究子目标，选择对应的嵌入技术：

| 子目标/信号关键词 | 嵌入技术 | 嵌入到哪种采集方式 |
|------------------|---------|------------------|
| 功能优先级、需求分类、先做什么 | **卡诺模型** | 嵌入到问卷调查中 |
| 体验诊断、整体体验水平、ETS | **ETS体验评估** | 嵌入到问卷调查中 |
| 用户选择、切换决策、购买动机、竞品对比 | **JTBD** | 嵌入到用户访谈中 |
| 信息架构、导航分类、内容归类 | **卡片分类** | 嵌入到用户访谈中 |
| 视觉注意、页面布局、视线 | **眼动测试** | 嵌入到可用性评估中 |
| 体验旅程、触点、流程 | **灵犀旅程** | 嵌入到用户访谈中 |
| 用户角色、群体特征、行为画像 | **用户画像** | 访谈+问卷合成综合产出 |

#### 2.3 合并同类阶段

检查是否可以合并：同一主采集方式承载多个嵌入技术。

**合并示例**：

| 用户需求 | 合并前（不合理） | 合并后（合理） |
|---------|---------------|-------------|
| 了解功能优先级+体验水平 | 阶段一:卡诺问卷 阶段二:ETS问卷 | 阶段一:问卷调查（含卡诺模块+ETS模块） |
| 需求探索+信息架构 | 阶段一:访谈 阶段二:卡片分类 | 阶段一:用户访谈（后半段嵌入卡片分类任务） |
| 可用性+视觉注意 | 阶段一:可用性 阶段二:眼动 | 阶段一:可用性评估（同步采集眼动数据） |

#### 2.4 读取参考文件

匹配完成后，读取对应方法的参考文件获取详细规则：

| 方法 | 参考文件 | 层级 |
|------|---------|------|
| 用户访谈 | `references/interview.md` | 数据采集方式 |
| 问卷调查 | `references/survey.md` | 数据采集方式 |
| 可用性评估 | `references/usability-test.md` | 数据采集方式 |
| 用户声音分析 | `references/voc-analysis.md` | 数据采集方式 |
| 卡诺模型 | `references/kano-model.md` | 嵌入技术 |
| ETS体验评估 | `references/ets-model.md` | 嵌入技术 |
| JTBD | `references/interview.md`（JTBD 章节） | 嵌入技术，依附用户访谈 |
| 卡片分类 | `references/card-sorting.md` | 嵌入技术 / 可独立执行 |
| 眼动测试 | `references/eye-tracking.md` | 嵌入技术 |
| 灵犀旅程 | `references/journey-mapping.md` | 嵌入技术/综合产出 |
| 用户画像 | `references/persona.md` | 综合产出 |

### Step 3: 方案设计

对每个研究阶段（每个阶段 = 一种数据采集方式 + 若干嵌入技术），设计以下要素：

#### 3.1 研究内容概览

**这是方案的核心新增模块**。为每个阶段描述具体研究什么内容，让研究团队拿到方案就知道访谈怎么问、问卷分几块、测试测什么。

**访谈类阶段的内容概览**——描述访谈话题模块和核心探针：

```
### 研究内容概览（用户访谈）
**主采集方式**：用户访谈
**嵌入技术**：卡片分类（后半段）

#### 模块一：认知画像（约 15 分钟）
- 核心探针：提到「存款产品」，您第一个想到的是什么？
- 了解用户如何自发描述和区分不同类型的存款产品
- 了解用户使用的分类词汇 vs 产品官方术语

#### 模块二：选择决策（约 20 分钟）
- 核心探针：您最近一次选存款产品是怎么决定的？看了哪些东西？
- 了解选择过程中的关键考虑因素及排序
- 了解信息获取渠道和信任来源

#### 模块三：使用体验与痛点（约 15 分钟）
- 核心探针：从第一次接触到现在的使用过程中，有没有什么让您觉得不方便或不满意的地方？
- 了解操作障碍、服务缺口和情绪低谷

#### 模块四：卡片分类任务（约 15 分钟，嵌入技术）
- 展示 30-40 张存款产品属性卡片
- 开放式分类：请用户按自己的理解自由分组并命名
- 记录出声思考，理解分组逻辑

#### 模块五：理想形态（约 10 分钟）
- 核心探针：如果可以自己设计一款存款产品，它应该是什么样的？
- 了解未满足的需求和期望
```

**问卷类阶段的内容概览**——描述问卷模块结构。注意：不应只列出嵌入技术相关的模块（如只写 ETS 3 题或 Kano 双题）。建议参考以下完整骨架，根据实际需求灵活增删辅助模块：

| 模块 | 内容 | 题型 | 题量 |
|------|------|------|------|
| 甄别题 | 产品持有情况、使用频率 | 单选 | 3-5题 |
| A.认知清晰度 | 对产品类型的区分能力 | 李克特5点 | 4-6题 |
| B.ETS体验评估 | NPS + 8维度双选题 (ETS-U) | NPS + 多选 | 3题 |
| C.痛点发生率 | 各项痛点的发生频率 | 多选+量表 | 5-8题 |
| ... | ... | ... | ... |
| 人口学 | 年龄/收入/城市 | 单选 | 4-5题 |
| 开放建议 | 最希望改进的方面 | 开放题 | 1-2题 |

辅助模块（甄别题、场景行为、人口学、开放建议等）**不是必须**，可根据总题量限制（≤25题、≤10分钟）和研究目的灵活增减。如果仅需快速扫描 NPS 趋势，ETS-U 的 3 题最小形态也合理——但需在方案中注明省略了哪些辅助模块及理由。
**主采集方式**：问卷调查
**嵌入技术**：ETS体验评估、卡诺模型

| 模块 | 内容 | 题型 | 题量 |
|------|------|------|------|
| 甄别题 | 产品持有情况、使用频率 | 单选 | 3-5题 |
| A.认知清晰度 | 对存款产品类型的区分能力 | 李克特5点 | 4-6题 |
| B.属性重要性 | 利率/期限/流动性等属性排序 | 矩阵排序 | 1题 |
| C.ETS体验评估 | NPS + 8维度双选题 (ETS-U) | NPS 0-10 + 多选 | 3题 |
| D.需求分类 | 各功能的正向+反向双题 (Kano) | Kano五点双题 | 每功能2题 |
| E.痛点发生率 | 各项痛点的发生频率 | 多选+量表 | 5-8题 |
| F.人口学 | 年龄/收入/城市/理财经验 | 单选 | 4-5题 |
```

**可用性评估阶段的内容概览**——描述测试任务：

```
### 研究内容概览（可用性评估）
**主采集方式**：可用性评估
**嵌入技术**：眼动测试（同步采集）

| 任务编号 | 任务场景 | 关键观察点 |
|---------|---------|-----------|
| T1 | 首次登录并完成身份验证 | 登录流程是否顺畅、验证方式是否清晰 |
| T2 | 查看最近一笔交易详情 | 信息查找效率、账单分类理解 |
| T3 | 发起一笔跨行转账 | 转账流程步骤数、金额输入体验 |
| T4 | 购买一款理财产品 | 产品信息可读性、购买决策支持 |
| T5 | 查找客服入口并联系在线客服 | 客服入口可见性、帮助信息有效性 |
```

#### 3.2 样本量设计

- 按主采集方式的经验值确定样本量范围
- 嵌入技术不增加额外样本量（与主采集方式共用）
- 参考文件中的样本量规则按主采集方式查阅
- 合并执行时以主采集方式的样本量为准

#### 3.3 样本特征要求

按主采集方式的要求设计筛选条件，从以下维度描述：
- **人口学特征**：年龄、性别、城市层级、收入等
- **行为特征**：产品使用频率、使用经验、使用场景
- **态度特征**：对产品的态度、付费意愿
- **排除条件**：行业从业者、竞品员工等

#### 3.4 样本配额要求

设计配额矩阵，维度 ≤ 3 个，按主采集方式的要求设计。

#### 3.5 实施要点

包含核心流程步骤、关键注意事项和物料准备。

#### 3.6 预期产出

列出该阶段将产出的成果物。

### Step 4: 样本量计算

默认使用主采集方式对应参考文件中的行业经验值。当用户提供置信水平等参数时，读取 `references/sample-size-calculator.md` 进行精确计算。

## 多阶段组合规则

1. **序贯原则**：定性在前（探索），定量在后（验证/测量）
2. **合并原则**：同一数据采集方式的嵌入技术应合并到同一阶段
3. **衔接说明**：明确前序阶段的产出如何作为后序阶段的输入
4. **样本独立性**：不同阶段招募不同用户
5. **阶段数 ≤ 4 个**，以数据采集方式计数

## 与下游生成技能的协作

### questionnaire-generator

当研究方案中包含问卷调查（含卡诺/ETS 等嵌入技术）时，在方案中标注详细问卷可通过 **questionnaire-generator** 技能生成。

### interview-guide-generator

当研究方案中包含用户访谈（含卡片分类/灵犀旅程等嵌入技术）时，在方案中标注详细访谈提纲可通过 **interview-guide-generator** 技能生成。

## 特殊场景处理

| 场景 | 处理方式 |
|------|---------|
| 用户没有明确目标 | 追问"你想了解哪方面的问题？目前的困惑是什么？" |
| 用户提供了已有数据/发现 | 在背景中引用，优先设计补充性研究 |
| 资源极度有限（1-2 天） | 推荐轻量方法（5 人快速访谈+嵌入卡片分类、在线问卷） |
| 涉及多个用户群体 | 按群体分别设计样本特征和配额 |
| B 端产品研究 | 样本量适当下调，注重决策链角色覆盖 |
| 需求超出 10 种方法范围 | 告知支持的 10 种方法，尝试替代或给出建议 |

## 质量检查清单

**方法层级匹配**
- [ ] 每个阶段是否以数据采集方式为锚点？
- [ ] 嵌入技术是否正确归入主采集方式阶段内？
- [ ] 是否有可以合并执行的阶段被拆开了？
- [ ] 阶段总数是否 ≤ 4（按数据采集方式计数）？

**研究内容概览**
- [ ] 每个访谈阶段是否输出了话题模块和核心探针？
- [ ] 每个问卷阶段是否输出了模块结构和大致题量？
- [ ] 问卷的内容概览是否考虑了嵌入技术之外的补充测量模块？如未包含，是否有合理的省略理由（如仅需快速扫描、时长限制等）？
- [ ] 每个可用性评估阶段是否输出了测试任务和观察要点？
- [ ] 嵌入技术的内容是否在主采集方式阶段内体现？

**样本设计**
- [ ] 样本量是否按主采集方式计算（嵌入技术不额外增加）？
- [ ] 样本特征是否具体可操作？
- [ ] 配额维度是否 ≤ 3？

**方案完整性**
- [ ] 是否输出了 JSON 和 Markdown 两种格式？
- [ ] 多阶段时是否标注了序贯关系？
- [ ] 是否说明了局限性？
